# Architecture

## Opening

This project is the first public version of `shomo.com` after a deliberate full reset. The previous branch state was stripped down so the new site could be designed without inheriting accidental structure, stale visuals, or a service model that no longer fit. What now exists is a document-first personal website built around an independent research practice. It still names prose, product, and productivity, but it no longer treats them as equivalent service lanes. Productivity is the research center. Prose and product are the crafts through which that inquiry becomes public.

The core problem the site solves is coherence. Someone arriving here needs to understand, quickly, how research, writing, product thinking, and systems work can belong to the same person without the site collapsing into a résumé dump or a service funnel. The architecture answers that by making the homepage behave like a public orientation layer. It establishes the practice, shows where the work lives, and only then explains how private work sometimes grows out of that public surface. The inner pages deepen each area without changing the overall tone.

Technically, the project is a Next.js App Router site written in TypeScript and styled with Tailwind plus a small amount of global CSS. Local content lives in MDX-shaped files under `content/`, which keeps most public copy editable without moving into a heavy CMS. Remote freshness is used only where it helps. Substack and YouTube feeds are imported as lightweight archives, but those imports are treated as optional. If the network fails, the manual public-work layer still carries the page.

## Ontology

The most important entity in the system is the site itself as a publishing and collaboration surface. It is not a general blog, not a project database, and not a service marketplace. Everything in the codebase serves that single surface.

Within that surface, `SiteSettings` in `src/lib/content-types.ts` and `src/lib/site-config.ts` defines the public identity of the project. It knows the site name, description, email address, navigation, social exits, and the remote feed sources the site is willing to trust. You can think of it as the diplomatic passport of the whole application. If the site needs to introduce itself to a browser, a crawler, or a visitor, this object is where the introduction begins.

`TrackPage` is the second core entity. The type name is historical shorthand for one of the three named areas on the site: prose, product, or productivity. Each one has a title, summary, list of what lives there, audience language, working ideas, a public-record heading, and a page-specific inquiry subject line. Even though the internal type still says “track,” the public site no longer presents these pages as service tracks. They are authored in `content/pages/` so the relationship between the research practice and its outputs can keep evolving without rewriting route logic.

`ProofItem` is the third major entity. The internal name remains `ProofItem`, but what it represents publicly is selected work: a public essay archive, a video channel, a code repository, or some other external signal that shows what the practice has actually produced. It has a medium, a set of areas it supports, a live URL, an optional metric, and a `hidden` state. That hidden state matters. The content model is allowed to know about things that are not yet ready for publication, but the interface is not allowed to render them. In other words, the ontology distinguishes between “known internally” and “fit for public display.”

`FeedSource` is the final named entity that shapes runtime behavior. A feed source describes a remote archive the site may consult for recent essays or videos. These sources are intentionally narrow. They are not a general ingestion layer for everything on the internet. They only exist so the Work page and the Prose page can show live archive context while still degrading gracefully if the feed is unavailable.

## Geography

At the repository root, the project is framed by both infrastructure and narrative. `package.json` and `package-lock.json` define the application and its dependencies. `tsconfig.json` defines how TypeScript understands aliases and framework types. `next.config.ts`, `tailwind.config.ts`, `postcss.config.mjs`, `.eslintrc.cjs`, `vitest.config.ts`, `vitest.setup.ts`, and `playwright.config.ts` describe how the system builds, styles, lints, and verifies itself. `.env.example` documents the public identity values that can be overridden at deploy time. `.gitignore` keeps generated material out of version control. `ARCHITECTURE.md` is the narrative map of the system, and `DEVLOG.md` is the running operational ledger.

The `content/` directory is the editable publishing layer. Inside `content/pages/`, `home.mdx`, `prose.mdx`, `product.mdx`, `productivity.mdx`, and `collaborate.mdx` define the public language of the homepage and inner pages. `home.mdx` frames the site as the documentation of an independent research practice. `prose.mdx` and `product.mdx` describe the two crafts through which that practice becomes public. `productivity.mdx` names the actual center of inquiry, and it is the only content page that speaks directly about AI. `collaborate.mdx` now functions as a short note on how private work happens around the public practice. These files are written as content artifacts rather than as code so the public framing can evolve independently. Inside `content/proof/`, each `.mdx` file represents one selected public item. `substack-essays.mdx`, `armchair-descending.mdx`, `linkedin-profile.mdx`, and `shomosite-repo.mdx` are live public signals. `private-systems-lab.mdx` is intentionally hidden, which lets the omission logic remain part of the real content model rather than a purely theoretical code path.

The `src/app/` directory is where the public route surface lives. `src/app/layout.tsx` is the root shell. It loads the local fonts from `src/app/fonts/`, imports `src/app/globals.css`, mounts the shared header and footer, and defines the default metadata base for the whole site. `src/app/page.tsx` is the homepage composition. `src/app/prose/page.tsx`, `src/app/product/page.tsx`, and `src/app/productivity/page.tsx` are the three main inner pages. `src/app/work/page.tsx` is the public work archive organized by medium. `src/app/collaborate/page.tsx` still owns the `/collaborate` route, but the page now presents itself publicly as “Working together.” `src/app/robots.ts`, `src/app/sitemap.ts`, and `src/app/icon.svg` exist for crawler behavior and browser identity rather than human reading.

The `src/components/` directory contains the reusable visual language of the site. `src/components/site-header.tsx` renders the persistent navigation shell, now including the softer “Working together” label. `src/components/site-footer.tsx` closes every page with the same quiet contact rhythm and the same description of the practice. `src/components/site-mark.tsx` holds the thin-line geometric mark that functions as the site’s small visual signature. `src/components/home-hero.tsx` renders the poster-like first viewport and points the first click toward `/work` rather than toward contact. `src/components/track-rail.tsx` stages the homepage’s handoff between prose, product, and productivity without turning them into an offer grid. `src/components/track-page-template.tsx` gives the inner pages a consistent narrative structure. `src/components/proof-list.tsx` and `src/components/archive-list.tsx` render manual public items and feed-backed archive rows without falling into card-grid habits. `src/components/rich-text.tsx` applies the typographic treatment for content loaded from Markdown. `src/components/section-reveal.tsx` owns the single reusable reveal motion used across the site.

The `src/lib/` directory is the behavioral backbone. `src/lib/content-types.ts` defines the shared contracts that hold the site together. `src/lib/site-config.ts` centralizes public settings and feed definitions. `src/lib/mailto.ts` standardizes inquiry links so every major CTA can prefill context cleanly. `src/lib/markdown.ts` converts Markdown content into HTML. `src/lib/content.ts` reads local content files, strips the narration comments that live at the top of MDX files, parses frontmatter, and exposes typed loaders for the routes. `src/lib/feeds.ts` parses and fetches Substack RSS and YouTube Atom archives with graceful fallbacks. `src/lib/seo.ts` builds metadata and JSON-LD. `src/lib/date.ts` formats public timestamps into the restrained editorial labels used in proof lists. `src/lib/classes.ts` exists only to keep conditional class strings from becoming noisy.

The `tests/` directory mirrors the architecture of confidence. `tests/unit/feeds.test.ts` verifies the pure parsing logic for external feeds. `tests/integration/home-page.test.tsx`, `tests/integration/track-routes.test.tsx`, and `tests/integration/work-page.test.tsx` render the real route components and check the narrative spine, route availability, omission logic, vocabulary constraints, AI-placement rules, and feed fallback behavior. `tests/e2e/navigation.spec.ts` runs a browser-level navigation walk through the primary information architecture. `playwright.config.ts` gives that browser test its own dedicated dev-server port so stale local servers cannot distort the result. The tests are not decorative. They correspond directly to the architectural promises of the site.

## Flow

The primary visitor flow begins at `/`. `src/app/page.tsx` loads the homepage content document, the working-together document, the three internal `TrackPage` documents, and the curated public items in parallel. It then renders them in a sequence that answers four questions. The hero establishes the practice at the highest level and sends the first click toward `/work`. The next section explains what the practice is. The rail answers where the work lives. The public-work section shows a selective body of published outputs. The closing note explains how private work may grow out of what is public here. That ordering is not incidental copy structure. It is the core runtime argument of the site.

The second flow begins when a visitor chooses one of the inner pages. The route loads one `TrackPage` document and the public items tagged for that area. In the case of `/prose`, it also requests the configured Substack and YouTube archives so the page can show live recent work alongside the curated manual layer. The route hands all of that into `src/components/track-page-template.tsx`, which keeps the same logic on every page: explain why the area exists, show what lives there, state who it matters to, present the working ideas underneath it, show the public record, and end with a quiet invitation to write. The page therefore behaves less like a marketing landing page and more like an organized extension of the public practice.

The third important flow runs through `/work`. This route loads all visible manual public items first, then fetches the remote archives in parallel. If the Substack or YouTube requests succeed, recent essays and videos appear as archive rows beneath the curated anchor items. If either request fails, the page does not collapse. Instead, `src/lib/feeds.ts` returns an error message and the UI offers the canonical channel or publication link. The important architectural decision here is that remote freshness enriches the page but never owns the page.

There is also a crawler flow. `src/app/robots.ts` and `src/app/sitemap.ts` generate search-facing outputs from the same shared settings and known route surface that the human UI uses. Metadata and JSON-LD are built from `src/lib/seo.ts`, which means search identity is derived from the same source of truth as the rest of the application rather than being managed separately.

## Philosophy

The strongest architectural decision in this codebase is that content and structure are kept close but not fused. Copy, selected public records, and the working-together note live in `content/`, where they remain editable as content. Layout, motion, and route assembly live in `src/`, where they remain code. This keeps the owner in control of the public language without requiring a CMS and without turning every copy change into a component rewrite.

The second principle is document-first restraint. The interface deliberately avoids card-first composition, decorative visual noise, image-led hero design, and aggressive sales language. The architecture supports that restraint by giving each component one clear job and by keeping the motion system small. `src/components/section-reveal.tsx` is enough to create presence without turning motion into a separate design ideology. The homepage rail creates the major handoff through layout rather than effect-heavy scripting.

The third principle is honest public work. The site is allowed to know about future items, but it is not allowed to publish them until they are real. That is why `ProofItem` has a `hidden` state and why the content loader filters hidden or URL-less entries out before rendering. It is also why live feeds are treated as optional enhancers. Public credibility should survive network failure.

Finally, the project treats narration as architecture rather than decoration. The MDX content files include file-level narration, the code files describe their role and dependencies, the devlog records the work that shaped the system, and this document explains the project as a coherent whole. The codebase is designed for someone who thinks in systems but reads in prose. That requirement is not adjacent to the implementation. It is part of the implementation.
